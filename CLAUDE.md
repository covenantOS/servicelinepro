# Website Factory - AI Instruction Set

This file is the brain of the Website Factory system. It contains everything an AI agent needs to know to build, maintain, and extend any site created from this template.

## System Overview

This is an Astro-based static site generator for local home service businesses. Each client site is produced by:

1. Cloning this repository
2. Filling in `site.config.yaml` with the client's business data
3. Generating all page content (markdown files in `src/content/`)
4. Adding client photos to `public/media/`
5. Building and deploying to Cloudflare Pages

The output is a 100-200 page static website with perfect PageSpeed scores, automatic internal linking, schema markup, and a client-facing CMS.

## Tech Stack

- **Astro** (latest stable) for static site generation. Zero JS shipped to browser.
- **Cloudflare Pages** for hosting. Free, global CDN, unlimited bandwidth.
- **Sveltia CMS** for client content editing (git-based, at `/admin/`).
- **Sharp** for image optimization to WebP.
- **Playwright** for visual QA and link checking.

## Project Structure

```
site.config.yaml          ← Central client data (name, services, areas, reviews, brand)
src/lib/config.ts         ← TypeScript config loader (import config from '../lib/config')
src/layouts/Base.astro    ← Master HTML shell, schema, meta, CSS custom properties
src/components/           ← All UI components (data-driven from config)
src/pages/                ← Page templates (index, about, contact, services/[slug], areas/[slug])
src/content/              ← Markdown content per service, area, and combo page
src/styles/global.css     ← Base styles, CSS custom properties, typography scale
public/media/             ← Client images (WebP format)
public/fonts/             ← Self-hosted Google Fonts (WOFF2)
public/admin/             ← Sveltia CMS entry point and config
scripts/                  ← Build and utility scripts
```

## Key Architecture Decisions

### Data Flow
Every component imports `config` from `src/lib/config.ts`. This reads `site.config.yaml` at build time. All business data, services, areas, reviews, trust signals, and brand settings come from this single file. No hardcoded client data in components.

### URL Structure
- Service pages: `/services/{service-slug}-{city}-{state}` (e.g., `/services/deck-building-topeka-ks`)
- Area pages: `/areas/{area-slug}` (e.g., `/areas/college-hill-topeka-ks`)
- Combo pages: `/services/{service-slug}-{city}-{state}/{area-slug}`
- Sub-service pages: `/services/{sub-slug}-{city}`

### Content System
Markdown files in `src/content/` provide page-specific content. Frontmatter contains: title, description, pricing tiers, FAQ questions, and process steps. The page templates load this content and inject it into the component framework.

Filename convention:
- Services: `src/content/services/{service-slug}.md`
- Areas: `src/content/areas/{area-slug}.md`
- Combos: `src/content/service-areas/{service-slug}-{area-slug}.md`

### CSS System
CSS custom properties defined in `global.css` and overridden by `site.config.yaml` values through the Base layout. Colors, fonts, spacing, shadows, and radii are all tokenized. Components use these tokens. Never hardcode colors.

### Typography Scale
Major Third (1.25 ratio): `--text-xs` through `--text-5xl`. Headings use `--font-heading` (Montserrat). Body uses `--font-body` (Inter). Both self-hosted as WOFF2.

## Content Generation Rules

### Writing Style (MANDATORY)
- NEVER use em dashes
- NEVER use: "best practices", "in today's world", "at the end of the day"
- NEVER use "X isn't just about Y" or "X is more than Y" structures
- NEVER use bait-and-switch sentences, juxtapositions, or contrasts
- NEVER use "merely" or "just" in a minimizing sense
- NO generic filler. Every sentence must contain specific, useful information
- Reference the business owner BY NAME throughout
- Reference specific neighborhoods with local knowledge
- Include real pricing ranges where appropriate
- Include specific material and product brand names
- Write like a human copywriter, not an AI

### SEO Requirements
- Every page: unique H1, unique title tag, unique meta description
- Service pages: 1500-2500 words
- Area pages: 800-1200 words with neighborhood-specific details
- Service-area combo pages: 600-1000 words
- Internal linking: service pages link to areas, area pages link to services, combo pages link to both parents and related pages
- Breadcrumbs on all inner pages
- FAQ section (4-6 questions) on every service page
- Schema markup: LocalBusiness (every page), Service (service pages), FAQPage (FAQ sections), BreadcrumbList (breadcrumbs)
- Sitemap auto-generated by @astrojs/sitemap

### Service Page Content Structure
1. H2 intro (2-3 paragraphs with owner name and local references)
2. H2 Materials/Options (detailed H3 subsections per material with pricing ranges)
3. H2 Permits and Building Codes (link to actual city permit office)
4. H2 Why Choose [Business Name] (specific to this service and company)
5. Sub-service cards grid (if applicable)
6. Pricing Guide section
7. Process Steps (numbered, 3-5 steps)
8. FAQ accordion (4-6 questions with FAQPage schema)
9. Areas We Serve links
10. CTA section

### Area Page Content Structure
1. H2 Neighborhood Overview (history, boundaries, architectural styles, character)
2. Services Available grid (cards linking to combo pages)
3. H2 Local Considerations (HOA rules, historic district codes, common home ages, weather)
4. Reviews snippet
5. CTA section
6. Nearby areas links

## Design Rules

### Anti-Vibe-Code Principles
1. NEVER use uniform border-radius. Vary by context (buttons 6px, cards 10px, images 0-4px).
2. NO gradient buttons. Solid colors with subtle hover darkening.
3. Left-aligned body text on content pages. Only center headings on homepage hero and short CTA sections.
4. Alternate layout patterns between sections. Never repeat the same layout twice in a row.
5. 60-30-10 color rule: 60% neutral, 30% dark anchors, 10% brand accent.
6. Vary section spacing: related sections 32-48px, distinct sections 64-80px.
7. Alternate heading treatment patterns (label+heading, heading+subtitle, heading+accent-line).
8. Trust signals woven throughout every page, not in one section.
9. Contact form or phone CTA within one scroll of any position.

### Responsive Breakpoints
- Mobile first. Base styles target mobile.
- Tablet: `@media (min-width: 768px)`
- Desktop: `@media (min-width: 1024px)`
- Wide: `@media (min-width: 1280px)`

## Common Tasks

### Adding a New Client
1. Clone this repo
2. Edit `site.config.yaml` with all client data
3. Replace placeholder images in `public/media/`
4. Generate content markdown files in `src/content/`
5. Update `public/admin/config.yml` backend repo reference
6. Build: `npm run build`
7. Deploy: `bash scripts/deploy.sh`

### Adding a New Service
1. Add the service to `site.config.yaml` under `services`
2. Create content file: `src/content/services/{slug}.md`
3. Add a hero image: `public/media/{slug}-hero.webp`
4. Create combo pages: `src/content/service-areas/{slug}-{area-slug}.md` for each area
5. Rebuild

### Adding a New Area
1. Add the area to `site.config.yaml` under `areas`
2. Create content file: `src/content/areas/{slug}.md`
3. Create combo pages for each service
4. Rebuild

### Image Handling
- All images must be WebP format
- Run `npm run optimize-images` to convert raw images
- Hero images: 1200px wide, quality 80-85
- Project photos: max 1600px wide
- Thumbnails: 400px wide
- Lazy-load everything below the fold
- Eager-load hero images and above-fold content

### Deployment
```bash
# Build and deploy to production
bash scripts/deploy.sh production

# Deploy to preview environment
bash scripts/deploy.sh preview
```

### Visual QA
```bash
# Start dev server, then in another terminal:
npx playwright install chromium
node scripts/visual-review.js http://localhost:4321
```

## Theme System

The `brand.theme` field in `site.config.yaml` determines design decisions per the industry profile specs. Current themes: `contractor`, `auto-shop`. Theme-specific CSS can be added to `src/styles/themes/`.

## Security

- Content Security Policy headers set via `_headers` file (created by deploy script)
- Honeypot field on all forms for bot protection
- No inline scripts from third parties without CSP allowance
- `/admin/` blocked from crawling via robots.txt
- No API keys in committed code (environment variables only)
